1. Introducción
1.1. Objetivo
El objetivo  de este documento  es proporcionar  una guía completa  para la integración  de aplicaciones  web con la plataforma  de inicio de 
sesión  único (SSO) de TYPSA.
La plataforma  TYPSA  SSO ofrece  servicios  centralizados  de autenticación  y autorización  mediante  la integración  de Microsoft  Azure 
Active Directory,  AWS Cognito  y mecanismos  de autenticación  específicos  de cada aplicación.
Esta guía describe  las implementaciones  de frontend  y backend  necesarias,  los flujos de autenticación,  los procesos  de gestión  de 
tokens,  las consideraciones  de seguridad  y los requisitos  operativos  necesarios  para integrar  con éxito una nueva aplicación  en el 
ecosistema  de autenticación  de TYPSA.
El documento  sirve  como  referencia  oficial  para  los equipos  de desarrollo  responsables  de implementar  la autenticación  y la 
autorización  en las aplicaciones  TYPSA.
1.2. Ámbito
Este  documento  abarca  el proceso  completo  de integración  entre  una  aplicación  cliente  y la plataforma  TYPSA  SSO.
El alcance  incluye:
▪ Configuración  y uso del servicio  TYPSA  SSO.
▪ Autenticación  a través de Microsoft  Azure Active Directory.
▪ Federación  a través de AWS Cognito.
▪ Requisitos  de implementación  del frontend.
▪ Requisitos  de implementación  del backend.
▪ Generación  y validación  de JWT de la aplicación.
▪ Mecanismos  de validación  y actualización  de tokens  de Cognito.
▪ Gestión  de cookies  compartidas.
▪ Controles  de seguridad  y mejores  prácticas  de autenticación.
▪ Solución de problemas y orientación operativa.
El enfoque  de integración  descrito  en este documento  es independiente  de la tecnología  y puede  implementarse  utilizando  cualquier  
marco  de trabajo  de frontend  o backend  capaz de admitir  la comunicación  HTTP,  el procesamiento  de JWT y el manejo  seguro  de 
cookies.
Algunos  ejemplos  son:
▪ Angular
▪ NestJS
▪ Express.js

2. Descripción general de la arquitectura
2.1. Arquitectura de autenticación
La plataforma  de inicio de sesión  único (SSO) de TYPSA  sigue un modelo  de autenticación  centralizado  que permite  a múltiples  
aplicaciones  autenticar  a los usuarios  a través de una infraestructura  común  de proveedor  de identidad.
La arquitectura  combina  Microsoft  Azure Active Directory,  AWS Cognito  y el servicio  TYPSA  SSO para proporcionar  una autenticación  
segura  y escalable  en múltiples  dominios  y aplicaciones  corporativas.
La solución  está diseñada  para admitir  escenarios  de autenticación  multitenant  en los que diferentes  organizaciones  se autentican  a 
través de sus propios  tenants  de Azure Active Directory,  al tiempo  que se mantiene  una experiencia  de autenticación  unificada.
La arquitectura  consta  de cuatro  componentes  principales:
Componente Responsabilidad
Navegador / Aplicación 
front-end
Inicia la autenticación,  procesa  las devoluciones  de llamada  y utiliza las API de la aplicación.
Servicio TYPSA SSO
Gestiona  la resolución  de inquilinos,  la redirección  de Azure AD, el intercambio  de tokens  y las 
operaciones  de actualización  de tokens.
Microsoft Azure Active 
Directory
Proporciona  autenticación  de usuarios  corporativos  y gestión  de identidades.
AWS Cognito
Actúa como capa de federación  y proveedor  de tokens  entre Azure AD y las aplicaciones  
TYPSA.
Backend de la aplicación
Valida  los tokens  de Cognito,  genera  JWT específicos  de la aplicación  y gestiona  las cookies  de 
sesión.
A continuación  se ilustra la arquitectura  de autenticación.
Diagrama de arquitectura
🡪  de usuario Interfaz de usuario🡪  Servicio TYPSA SSO🡪  Azure Active Directory🡪 AWS Cognito🡪  Backend de la aplicación🡪  API  protegidas
2.2. Resumen del flujo de autenticación
La autenticación  se lleva a cabo mediante  una secuencia  de interacciones  coordinadas  entre la aplicación  front-end,  el servicio  TYPSA  
SSO, Azure Active Directory,  AWS Cognito  y el backend  de la aplicación.
El proceso  comienza  cuando  un usuario  introduce  una dirección  de correo  electrónico  corporativa  y continúa  hasta que se ha emitido  un 
token de autorización  específico  de la aplicación.
A continuación  se describe  el flujo de autenticación  a alto nivel.

Paso 1: solicitud de inicio de sesión del usuario
El usuario accede a la página de inicio de sesión de la aplicación e introduce una dirección de correo electrónico 
corporativa. La aplicación front-end reenvía la solicitud al servicio TYPSA SSO.
Paso 2: resolución del inquilino
El servicio  TYPSA  SSO analiza  el dominio  de correo  electrónico  e identifica  el inquilino  de Azure Active Directory  correspondiente.
Si el inquilino  no está registrado  en la plataforma,  se rechaza  la autenticación
Paso 3: Autenticación de Azure AD
El usuario  es redirigido  a la página  de inicio de sesión  de Microsoft  Azure Active Directory.
Azure AD valida las credenciales  corporativas  del usuario  y aplica los controles  de seguridad  configurados,  tales como:
▪ Autenticación  multifactor  (MFA)
▪ Políticas  de acceso  condicional
▪ Verificación  del cumplimiento  de los dispositivos
▪ Políticas  de seguridad  corporativas
Paso 4: emisión del token de Cognito
Tras una autenticación  correcta,  Azure AD devuelve  un código  de autorización  a AWS Cognito.  AWS Cognito  
intercambia  el código  de autorización  y genera  los siguientes  tokens:
Token Finalidad
Token de identificación Contiene  las reclamaciones  de identidad  del usuario.
Token  de acceso Otorga  acceso  a los recursos  protegidos  de Cognito.
Token de actualización Permite  renovar  el token sin necesidad  de volver a autenticarse.
Paso 5: Procesamiento de la llamada de retorno del frontend
El servicio  TYPSA  SSO redirige  el navegador  de vuelta a la ruta de callback  de la aplicación.  La URL de callback  
contiene  los tokens  de Cognito  necesarios  para la siguiente  fase de autenticación.  El frontend  extrae  estos 
valores  y prepara  la solicitud  de inicio de sesión  de la aplicación.
Paso 6: Autenticación de la aplicación
El frontend envía los tokens de Cognito al backend de la aplicación utilizando el punto final de inicio de sesión SSO dedicado.

El backend  valida la identidad  de Cognito  y recupera  o crea la cuenta  de usuario  correspondiente  dentro  de la aplicación.
Paso 7: Generación del token de la aplicación
Tras una validación  satisfactoria,  el backend  genera  un JWT específico  de la aplicación  que contiene  información  de autorización,  como:
▪ Identificador de usuario
▪ Nombre
▪ Dirección de correo electrónico
▪ Roles asignados
▪ Identificador de sesión
Este token es independiente  del token de Cognito  y se utiliza para la autorización  de la aplicación.
Paso 8: Establecimiento de la sesión
El backend  crea las cookies  de autenticación  TYPSA  necesarias  y devuelve  el JWT de la aplicación  al frontend.
El frontend  almacena  el token  de la aplicación  e inicia  la comunicación  autenticada  con  los servicios  protegidos  de la aplicación.
Resumen del flujo de identidad
El proceso  completo  de autenticación  se puede  resumir  de la siguiente  manera:
1. El usuario introduce su correo electrónico corporativo.
2. El frontend redirige al servicio SSO de TYPSA.
3. Se identifica al inquilino mediante el dominio de correo electrónico.
4. Azure Active Directory autentica al usuario.
5. AWS Cognito emite tokens de autenticación.
6. El frontend recibe los tokens de Cognito.
7. El backend valida la identidad de Cognito.
8. El backend  genera  el JWT de la aplicación.
9. Se crean cookies de autenticación.
10. El usuario obtiene acceso a los recursos protegidos.

3. Servicio SSO
3.1. Descripción general del servicio
El servicio  de inicio de sesión  único (SSO) de TYPSA  actúa como puerta  de enlace  de autenticación  central  para todas las aplicaciones  
integradas  en el ecosistema  TYPSA.
El servicio  se encarga  de gestionar  las solicitudes  de autenticación,  la resolución  de inquilinos,  la federación  de Azure Active Directory,  el 
intercambio  de tokens  de AWS Cognito,  las operaciones  de actualización  de tokens  y la recuperación  de perfiles  de usuario.  Al centralizar  
las responsabilidades  de autenticación,  la plataforma  garantiza  una experiencia  de usuario  coherente,  una incorporación  simplificada  de 
aplicaciones  y una mayor  seguridad  en todas las aplicaciones  de TYPSA.
El servicio  SSO se implementa  como una solución  sin servidor  alojada  en AWS y expuesta  a través de Amazon  API Gateway.  El servicio  
admite  múltiples  inquilinos  de Azure Active Directory  y permite  a las organizaciones  autenticarse  utilizando  sus credenciales  corporativas  
existentes.
Los objetivos  principales  del servicio  SSO son:
▪ Centralizar  la autenticación  de usuarios.
▪ Admitir  la integración  de Azure Active Directory  multitenant.
▪ Simplificar  la incorporación  de aplicaciones.
▪ Proporcionar  una gestión  segura  de tokens.
▪ Estandarizar  los flujos de autenticación  en todas las aplicaciones.
▪ Reducir  el esfuerzo  de implementación  relacionado  con la autenticación  para los equipos  de desarrollo.
Responsabilidades clave
El servicio  SSO realiza  las siguientes  funciones:
Responsabilidad Descripción
Resolución de inquilinos
Identifica  el inquilino  correcto  de Azure Active Directory  en función  del dominio  de correo  electrónico  del 
usuario.
Enrutamiento de 
autenticación
Redirige  a los usuarios  al punto final de autenticación  de Azure AD adecuado  a través de AWS Cognito.
Intercambio de tokens Intercambia  códigos  de autorización  por tokens  de autenticación  de Cognito.
Actualización de tokens Genera  nuevos  tokens  de Cognito  cuando  caducan  los tokens  de acceso.
Recuperación  del perfil de 
usuario
Recupera  la información  del perfil de usuario  y las fotos de perfil de Microsoft  Graph.
Incorporación de clientes Admite  los procesos  de registro  y consentimiento  para nuevas  organizaciones.

Arquitectura del servicio
El servicio  SSO se implementa  utilizando  los siguientes  componentes  de AWS:
Componente Finalidad
AWS Lambda Ejecuta  la lógica de autenticación  y gestión  de tokens.
Amazon API Gateway Expone puntos finales REST públicos.
AWS Cognito Proporciona  federación  de identidades  y emisión  de tokens.
MongoDB Almacena  la configuración  de los inquilinos  y las asignaciones  de dominios.
Microsoft  Azure AD Realiza  la autenticación  de usuarios  y la validación  de identidades.
Posición de integración
Dentro  de la arquitectura  de autenticación,  el servicio  SSO actúa como intermediario  entre las aplicaciones  cliente  y los proveedores  de 
identidad  externos.
El servicio  SSO no emite tokens  de autorización  específicos  de la aplicación.  En su lugar, proporciona  tokens  de autenticación  de Cognito  
que posteriormente  son validados  por el backend  de la aplicación,  el cual genera  su propio  JWT de aplicación  con fines de autorización.
3.2. Configuración del entorno
Antes de integrar  una aplicación  con la plataforma  SSO de TYPSA,  se debe configurar  el punto final SSO adecuado  según el entorno  de 
implementación.
El punto final de SSO es utilizado  tanto por los componentes  frontend  como por los backend  durante  las operaciones  de autenticación  y 
actualización  de tokens.
Configuración del frontend
Las aplicaciones  front-end  deben  almacenar  el punto final de SSO dentro  de su configuración  de entorno.

3.3. Puntos finales disponibles
El servicio  TYPSA  SSO expone  varios puntos  finales  REST que admiten  la autenticación,  la gestión  de tokens,  la recuperación  de perfiles  
de usuario  y la administración  de inquilinos.
La siguiente  tabla resume  los puntos  finales  disponibles  actualmente.
Método Punto final Descripción
GET /sso Inicia el proceso  de autenticación.
GET /sso/callback Punto final interno  de callback  de OAuth.
POST /sso/refresh-token Actualiza  los tokens  de autenticación  de Cognito.
GET /sso/m e Recupera  la foto de perfil del usuario  autenticado.
GET /sso/me/profile Recupera  la información  del perfil del usuario.
GET /sso/tenant-register Inicia el registro  del inquilino.
GET /sso/tenant-callback Gestiona  la llamada  de retorno  del registro  del inquilino.
3.4. Inicio de la autenticación (GET /sso)
El punto final de inicio de la autenticación  es el punto de entrada  al proceso  de inicio de sesión  único (SSO) de TYPSA.  Se encarga  de 
identificar  la organización  del usuario,  resolver  el inquilino  de Azure Active Directory  correspondiente  y redirigir  al usuario  al proveedor  de 
autenticación  adecuado  a través de AWS Cognito.
Normalmente,  la aplicación  frontend  invoca  este punto final después  de que el usuario  introduzca  una dirección  de correo  electrónico  
corporativa  en la página  de inicio de sesión.
El punto final no realiza  la autenticación  directamente.  En su lugar, prepara  la solicitud  de autenticación  y redirige  al usuario  al proveedor  
de identidad  responsable  de validar  las credenciales  del usuario.

A diferencia  de los encabezados  de solicitud,  el valor de la respuesta  no incluye  el prefijo Bearer.
Responsabilidades del frontend
Cuando  se recibe un nuevo encabezado  de autorización:
1. Leer el encabezado.
2. Sustituir  el token existente.
3. Actualizar  el almacenamiento  local.
4. Utilizar  el nuevo token para futuras  solicitudes.
Este proceso  debe ser gestionado  automáticamente  por el interceptor  HTTP.

12. Implementaciones de referencia
12.1. Descripción general
Esta  sección  proporciona  patrones  de implementación  de referencia  y enfoques  de arquitectura  recomendados  para  aplicaciones  que  
se integran  con  la plataforma  de inicio  de sesión  único  TYPSA.
El objetivo  no es prescribir  una pila tecnológica  específica,  sino proporcionar  patrones  de implementación  probados  que se ajusten  a la 
arquitectura  de autenticación  descrita  a lo largo de este documento.
Los  ejemplos  presentados  en esta  sección  se basan  en implementaciones  que  se utilizan  actualmente  en el ecosistema  TYPSA  y 
pueden  adaptarse  para  cumplir  con  los requisitos  específicos  de cada  proyecto.
12.2. Arquitectura de autenticación recomendada
Las aplicaciones  deben  implementar  una arquitectura  de autenticación  por capas que separe  la gestión  de identidades  de la autorización  de 
aplicaciones.

AWS Cognito Gestión de federación y tokens
API de backend Autorización  y gestión  de sesiones

13. Glosario
13.1. Descripción general
Este glosario  define los términos  clave, las tecnologías,  las siglas y los conceptos  utilizados  a lo largo de este documento.
El objetivo  de esta sección  es proporcionar  una comprensión  común  de la terminología  de autenticación  y los conceptos  específicos  de la 
plataforma  utilizados  en el ecosistema  de inicio de sesión  único (SSO) de TYPSA.
13.2. Términos de autenticación
Término Definición
Autenticación Proceso  de verificación  de la identidad  de un usuario.
Autorización Proceso  de determinar  a qué tiene permiso  de acceso  un usuario  autenticado.
Proveedor  de identidad  (IdP) Sistema  responsable  de autenticar  a los usuarios  y proporcionar  información  de identidad.
Sesión Interacción  temporal  autenticada  entre un usuario  y una aplicación.
Inicio de sesión  único (SSO)
Mecanismo  de autenticación  que permite  a los usuarios  acceder  a múltiples  aplicaciones  utilizando  un 
único conjunto  de credenciales.
Federación Proceso  de confiar  en un proveedor  de identidad  externo  para la autenticación.
Inquilino Espacio  de identidad  organizativo  aislado  dentro  de Azure Active Directory.
Dominio
El dominio  de correo  electrónico  utilizado  para identificar  una organización  y determinar  el proveedor  de 
autenticación  adecuado.
13.3. Terminología de los tokens
Término Definición
Token  de acceso Un token utilizado  para autorizar  el acceso  a recursos  protegidos.
Token de identificación Un token que contiene  información  sobre el usuario  autenticado.
Token de actualización
Token utilizado para obtener nuevos tokens de autenticación sin que el usuario tenga que volver a iniciar 
sesión.
JWT (JSON  Web Token)
Formato  de token compacto  que se utiliza para intercambiar  información  de forma segura  entre 
sistemas.
Caducidad del token La fecha y hora a partir de las cuales  un token deja de ser válido.
Actualización del token Proceso  de obtención  de nuevos  tokens  de autenticación  utilizando  un token de actualización  válido.
Validación del token Proceso  de verificación  de la integridad,  autenticidad  y caducidad  del token.
Reclamaciones Información  almacenada  en la carga útil de un JWT.

13.4. Términos de la plataforma TYPSA
Término Definición
Servicio SSO de TYPSA El servicio  de autenticación  centralizado  utilizado  en todas las aplicaciones  de TYPSA.
JWT de la aplicación
Un token de autorización  generado  por el backend  de una aplicación  tras una validación  correcta  de 
Cognito.
Correo  electrónico  activo La dirección  de correo  electrónico  asociada  al usuario  actualmente  autenticado.
safeEmail
Una representación  codificada  de forma segura  para URL de la dirección  de correo  electrónico  de un 
usuario,  utilizada  en las convenciones  de nomenclatura  de cookies.
Cookie de autenticación
Una cookie  utilizada  para almacenar  información  relacionada  con la autenticación.
Sesión  de la aplicación Un estado  de autenticación  establecido  entre el usuario  y una aplicación.
13.5. Términos de Microsoft Azure Active Directory
Término Definición
Azure Active Directory 
(Azure AD) Plataforma  de gestión  de identidades  y accesos  basada  en la nube de Microsoft.
Microsoft  Entra ID La nueva marca  de Azure Active Directory.
Acceso  condicional Directivas  que controlan  el acceso  en función  del usuario,  el dispositivo,  la ubicación  o el riesgo.
Autenticación  multifactor  (MFA)
Autenticación  que requiere  varios métodos  de verificación.
Administrador de inquilinos Usuario  responsable  de gestionar  la configuración  y los permisos  del inquilino  de Azure AD.
Aplicación  empresarial Aplicación  registrada  en Azure AD con fines de autenticación.
Consentimiento  del administrador
Autorización  concedida  por un administrador  de inquilino  que permite  a una aplicación  acceder  a 
los recursos  de la organización.
13.6. Términos de AWS Cognito
Término Definición
AWS Cognito Servicio  de gestión  de identidades  y accesos  de Amazon  Web Services.
Grupo  de usuarios Un directorio  de usuarios  gestionado  por Cognito.
Proveedor  de identidad  (IdP) Proveedor  de autenticación  externo  conectado  a Cognito.
Interfaz  de usuario  alojada Interfaz  de autenticación  proporcionada  por Cognito.
Flujo de código  de autorización Flujo OAuth  utilizado  para obtener  tokens  de autenticación  de forma segura.
REFRESH_TOKEN_AUTH Flujo de autenticación  de Cognito  utilizado  para actualizar  tokens.

13.7. Términos de seguridad
Término Definición
HTTPS Versión  segura  de HTTP que cifra el tráfico de red.
Cookie  segura Cookie  transmitida  únicamente  a través de conexiones  HTTPS.
Cookie solo HTTP Cookie  inaccesible  para el JavaScript  del navegador.
Cookie  SameSite Atributo  de cookie  que controla  el comportamiento  de la transmisión  entre sitios.
CSRF (Falsificación de solicitudes entre sitios) Ataque  que engaña  al usuario  para que realice  acciones  no deseadas.
Secuestro de sesión Apropiación  no autorizada  de una sesión  autenticada.
Principio  del mínimo  privilegio Principio  de seguridad  que consiste  en conceder  únicamente  los permisos  mínimos  
necesarios.
Defensa  en profundidad Estrategia  de seguridad  que utiliza múltiples  capas de protección.
13.8. Resumen
Este glosario  sirve como referencia  para la terminología  utilizada  a lo largo de este documento  y debe consultarse  siempre  que se 
requiera  una aclaración  sobre conceptos  específicos  de autenticación,  autorización,  Azure Active Directory,  AWS Cognito  o TYPSA.
Una comprensión  común  de estos términos  ayuda a garantizar  una implementación,  comunicación  y mantenimiento  coherentes  de las 
aplicaciones  integradas  con la plataforma  de inicio de sesión  único de TYPSA.
